# 🚀 راهنمای CI/CD پروژه کولیب

## 📖 مقدمه

در پروژه کولیب تمامی فرآیند Build، Test و انتشار Docker Image به صورت خودکار توسط **GitHub Actions** انجام می شود.

هدف این Pipeline این است که:

* ✅ از سالم بودن پروژه قبل از انتشار اطمینان حاصل شود.
* 🧪 تمامی تست های پروژه اجرا شوند.
* 🐳 تنها در صورت موفق بودن تست ها، Docker Image جدید ساخته شود.
* 📦 آخرین نسخه Image در Docker Hub منتشر شود.
* ⚡ فرآیند انتشار کاملا خودکار، قابل تکرار و بدون دخالت دستی باشد.

این ساختار مطابق با Best Practice های پروژه های پروداکشن طراحی شده است.


# 🏗 مراحل اجرای Pipeline

با هر Push یا Pull Request روی Branch `main`، GitHub Actions مراحل زیر را اجرا می کند.

## 1️. دریافت سورس کد

در ابتدا Runner آخرین نسخه Repository را دریافت می کند.

```
actions/checkout
```

در این مرحله دقیقا همان کدی که روی GitHub قرار گرفته است در Runner اجرا خواهد شد.


## 2️. ایجاد فایل تنظیمات

به دلایل امنیتی فایل `.env` داخل Repository قرار نمی گیرد.

در زمان اجرای Pipeline، GitHub Actions فایل `.env` را به صورت خودکار ایجاد می کند و مقادیر آن را از **GitHub Secrets** دریافت می کند.

## 3️. ساخت Docker Image ها

سپس Docker Compose اجرا می شود.

```
docker compose build
```

در این مرحله تمامی سرویس ها Build می شوند.

مانند:

* Django
* PostgreSQL
* Redis
* Elasticsearch
* MinIO
* Mailpit

## 4️. اجرای سرویس ها

پس از Build شدن Image ها، تمامی Container ها اجرا می شوند.

```
docker compose up -d
```

Runner دقیقا همان محیطی را اجرا می کند که در Production استفاده خواهد شد.

## 5️. انتظار برای آماده شدن جنگو

ممکن است PostgreSQL یا Elasticsearch هنوز آماده نباشند.

به همین دلیل Pipeline منتظر می ماند تا دستور زیر بدون خطا اجرا شود.

```
python manage.py check
```

این کار از Fail شدن تصادفی Pipeline جلوگیری می کند.

## 6️. ساخت Elasticsearch Index

قبل از اجرای تست ها، Index های Elasticsearch ساخته می شوند.

```
python manage.py search_index --rebuild -f
```

در نتیجه محیط تست کاملا آماده خواهد بود.

## 7️. اجرای تمامی تست ها

در این مرحله کل تست های Django اجرا می شوند.

```
python manage.py test
```

این قسمت مهم ترین مرحله Pipeline است.

در صورت وجود هرگونه خطا:

* Build متوقف می شود.
* Docker Image منتشر نمی شود.
* Workflow با وضعیت Failure پایان می یابد.

## 8️. نمایش Log ها در صورت خطا

اگر هر مرحله با شکست مواجه شود، لاگ تمامی کانتینر ها چاپ خواهد شد.

```
docker compose logs
```

این موضوع دیباگ کردن Pipeline را بسیار ساده تر می کند.

## 9️. انتشار Docker Image

اگر تمام مراحل قبلی با موفقیت انجام شوند:

* 🔑 ورود به Docker Hub انجام می شود.
* 🏗 Image جدید Build می شود.
* 🏷 تگ های مناسب ایجاد می شوند.
* ☁️ Image در Docker Hub منتشر می شود.

نمونه Tag ها:

```
latest
```

```
sha-8f31a4d
```

این ساختار باعث می شود همیشه علاوه بر آخرین نسخه، نسخه دقیق مربوط به هر Commit نیز قابل استفاده باشد.

## 10. پاک سازی محیط اجرا

در پایان تمامی کانتینر ها حذف می شوند.

```
docker compose down -v
```

این مرحله باعث می شود Runner برای اجرای بعدی کاملا تمیز باشد.

# 🔐 مدیریت Secret ها

تمام اطلاعات حساس پروژه از طریق GitHub Secrets مدیریت می شوند.

نمونه هایی از Secret ها:

* SECRET_KEY
* POSTGRES_PASSWORD
* REDIS_PASSWORD
* OPENROUTER_API_KEY
* MINIO_ROOT_PASSWORD
* Docker Hub Username
* Docker Hub Access Token

به این ترتیب هیچ اطلاعات حساسی داخل Repository قرار نمی گیرد.


# 🐳 فرآیند انتشار Docker Image

انتشار Image تنها زمانی انجام می شود که:

* ✅ Build موفق باشد.
* ✅ تمامی تست ها Pass شوند.
* ✅ تمامی سرویس ها بدون خطا اجرا شوند.

در غیر این صورت هیچ نسخه جدیدی در Docker Hub منتشر نخواهد شد.

این موضوع از انتشار نسخه های معیوب جلوگیری می کند.

# 📦 نسخه بندی Image ها

هر Build دو Tag دریافت می کند.

### آخرین نسخه

```
latest
```

### نسخه متناظر با Commit

```
sha-<commit-hash>
```

مزایا:

* امکان بازگشت به نسخه های قبلی
* رهگیری دقیق هر نسخه
* تاریخچه کامل انتشار Image ها


# 🎯 مزایای این Pipeline

* ⚡ اجرای کاملا خودکار
* 🧪 تست قبل از انتشار
* 🔒 امنیت بالا با GitHub Secrets
* 🐳 انتشار خودکار Docker Image
* 📈 قابلیت مقیاس پذیری بالا
* 🧹 پاک سازی خودکار Runner
* 🛡 جلوگیری از انتشار نسخه های خراب
* 🔄 قابل اجرا برای هر Commit
* 📦 سازگار با Docker Hub
* 🏭 مناسب برای محیط Production


# 🔮 مسیر توسعه آینده

در مراحل آینده میتوان CI/CD کولیب را به یک روند کاملا خودکار تبدیل کرد

به عنوان مثال پس از انتشار موفق Image:

* اتصال امن به سرور از طریق SSH
* اجرای `docker pull`
* اجرای `docker compose up -d`
* راه اندازی مجدد سرویس ها
* بررسی سلامت سرویس ها (Health Check)
* ارسال اعلان موفقیت یا خطا به Telegram، Discord یا Slack
